# Kensington 트랙볼 매핑 도구 — 작업 정리

> Pop!_OS 24.04 (COSMIC, Wayland)에서 Kensington 트랙볼 버튼을 브라우저 동작 등으로
> 매핑하는 커스텀 도구 제작 + 트랙볼 연결 문제 진단 기록.
> 최종 업데이트: 2026-05-31

---

## 1. 목표

Kensington 트랙볼용 Linux 공식 드라이버(KensingtonWorks)가 없어서, 트랙볼 버튼을
브라우저 뒤로/앞으로 등 다양한 동작에 매핑하는 도구를 직접 제작.

- 우선 기능: **브라우저 뒤로/앞으로** (Alt+Left / Alt+Right)
- 확장 가능: 임의 키조합, 명령 실행, 볼 스크롤, 정밀 모드

## 2. 환경

| 항목 | 값 |
|------|-----|
| OS | Pop!_OS 24.04 LTS |
| 데스크톱 | COSMIC |
| 세션 | **Wayland** |
| 트랙볼 | Kensington Expert Wireless Trackball (K72359WW), USB `047d:8018` |
| (보유) | Kensington Slimblade Trackball, USB `047d:2041` |

**핵심 제약:** Wayland라서 X11 도구(`xinput`/`xdotool`)는 사용 불가.
→ 커널 레벨 `evdev`(입력 읽기) + `uinput`(이벤트 주입) 방식으로 구현.
이 방식은 compositor와 무관하게 동작한다.

## 3. 설계 / 동작 방식

1. 트랙볼의 포인터 event 디바이스를 **독점 grab**(`EVIOCGRAB`).
2. 가상 `uinput` 디바이스를 만들어 일반 커서 이동 / 스크롤은 그대로 전달.
3. 설정된 물리 버튼은 클릭 대신 **액션**으로 치환:
   - `passthrough` — 일반 클릭
   - `key` — 키조합 주입 (예: Alt+Left = 뒤로). 커널 레벨이라 모든 앱/Wayland에서 동작
   - `command` — 프로그램 실행 (데스크톱 사용자 권한으로)
   - `scroll_hold` — 누르는 동안 볼 굴리면 스크롤
   - `precision_hold` — 누르는 동안 커서 속도 감소
4. systemd 서비스로 root 실행(= `/dev/input` 읽기 + `/dev/uinput` 쓰기), 부팅 시 자동 시작.
5. 무선 재연결 시 자동으로 디바이스를 다시 잡음.

### 디바이스 매칭 (USB·블루투스 모두 인식)

config의 `device_match`(이름 부분일치, 대소문자 무시)로 매칭.
→ **같은 트랙볼을 USB 나노 리시버로 연결하든 블루투스로 연결하든** 자동 인식.
→ 여러 트랙볼(Expert / Slimblade) 동시 커버.

## 4. 파일 구성 (`~/kensington-trackball/`)

| 파일 | 설명 |
|------|------|
| `trackball_mapper.py` | 데몬 + CLI (`run` / `learn` / `list` / `check-config`) |
| `config.toml` | 디바이스 매칭 + 버튼 매핑 설정 |
| `ktrackball.service` | systemd 자동시작 유닛 |
| `install.sh` | 설치 스크립트 (sudo 필요) |
| `README.md` | 사용 설명 |
| `작업정리.md` | 이 문서 |

### 설치 위치 (install.sh 실행 후)
- 데몬: `/opt/ktrackball/trackball_mapper.py`
- 설정: `/etc/ktrackball/config.toml`
- 유닛: `/etc/systemd/system/ktrackball.service`

## 5. 주요 명령

```bash
# 설치
cd ~/kensington-trackball && sudo ./install.sh

# 코드/설정 갱신 후 재배포
sudo cp ~/kensington-trackball/trackball_mapper.py /opt/ktrackball/
sudo install -m644 ~/kensington-trackball/config.toml /etc/ktrackball/config.toml
sudo systemctl restart ktrackball

# 연결된 입력장치 + 이름 확인
python3 /opt/ktrackball/trackball_mapper.py list

# 어떤 물리 버튼이 어떤 코드(BTN_SIDE 등)를 내는지 확인 (서비스 먼저 중지)
sudo systemctl stop ktrackball
sudo python3 /opt/ktrackball/trackball_mapper.py learn \
     --device /dev/input/by-id/usb-Kensington_Expert_Wireless_TB-event-mouse

# 설정 검증 / 로그
python3 /opt/ktrackball/trackball_mapper.py check-config --config /etc/ktrackball/config.toml
journalctl -u ktrackball -f
```

> 주의: `--config`는 서브명령 앞/뒤 어디 와도 되도록 수정됨(초기 버전은 앞에만 허용).

## 6. config.toml 현재 내용 요지

```toml
device_match = ["Expert Wireless", "Slimblade"]   # USB·BT·두 트랙볼 모두 매칭

[buttons]
BTN_LEFT  = { type = "passthrough" }
BTN_RIGHT = { type = "passthrough" }
BTN_SIDE  = { type = "key", keys = ["KEY_LEFTALT", "KEY_LEFT"]  }   # 뒤로
BTN_EXTRA = { type = "key", keys = ["KEY_LEFTALT", "KEY_RIGHT"] }   # 앞으로
# (scroll_hold / precision_hold / command 예시는 주석 처리됨)
```

> `BTN_SIDE`/`BTN_EXTRA`가 실제로 어느 물리 버튼인지는 `learn`으로 확정 필요.

## 7. ⚠️ 현재 막힌 문제 — 트랙볼 USB 연결 불량 (미해결)

소프트웨어 도구는 완성됐으나, **트랙볼이 USB에서 안정적으로 인식되지 않아** 실제 테스트를
못 한 상태. **소프트웨어가 아닌 하드웨어/연결 문제.**

### 증상 (커널 로그, 매번 동일하게 반복)
```
20:06:27  usb 1-1: new full-speed USB device number 34
20:06:28  Product: Expert Wireless TB  (정상 인식됨)
20:06:30  usb 1-1: USB disconnect, device number 34      ← 약 3초 만에 끊김
20:06:32  usb1-port1: Cannot enable. Maybe the USB cable is bad?
20:06:35  usb1-port1: unable to enumerate USB device     ← 재열거 실패
```
- 인식 → **~3초 후 끊김** → 재열거 실패가 반복.
- port1, port2 양쪽에서 동일 → 리시버 쪽 공통 원인 가능성.
- 핫플러그 메커니즘 자체는 정상 (꽂으면 즉시 인식은 됨). 부팅 때만 되는 구조 아님.

### 유력 가설
1. **USB autosuspend** (`/sys/module/usbcore/parameters/autosuspend = 2`, 2초).
   "3초 후 끊김" 타이밍과 일치. 일부 무선 리시버가 USB 절전 복귀에 실패.
2. **리시버/포트 하드웨어 불량.** 커널이 직접 "cable is bad / unable to enumerate" 보고.

### 다음에 할 일 (진단 분기)

**[1단계] autosuspend 끄고 테스트**
```bash
echo -1 | sudo tee /sys/module/usbcore/parameters/autosuspend
# → 리시버 뽑았다 다시 꽂기
sleep 5; lsusb | grep -i 047d && echo 유지됨 || echo 여전히끊김
journalctl -k -b --no-pager | grep -iE "usb 1-|enumerate|disconnect" | tail -8
```

- **유지되면 →** autosuspend가 원인. 아래 udev 규칙으로 영구 적용 후 매핑 진행.
  ```
  # /etc/udev/rules.d/50-kensington-noautosuspend.rules
  ACTION=="add", SUBSYSTEM=="usb", ATTR{idVendor}=="047d", ATTR{idProduct}=="8018", TEST=="power/control", ATTR{power/control}="on"
  ```
  적용: `sudo udevadm control --reload && sudo udevadm trigger`

- **여전히 3초 후 끊기면 →** 하드웨어 불량 확정.
  - 리시버를 **다른 PC**에 꽂아 동작 확인 (리시버 사망 여부 판별)
  - 포트에 **다른 USB 장치** 꽂아 포트 정상 여부 확인
  - 또는 **블루투스로 연결** (리시버 우회). 코드는 BT 이름도 부분일치로 자동 인식.
    ```
    bluetoothctl → scan on → pair <MAC> → trust <MAC> → connect <MAC>
    ```

### 연결 성공 후 마무리 절차
1. 코드/설정 재배포 + 서비스 재시작 (위 5절 명령).
2. `journalctl -u ktrackball -n 5` 에 `opened /dev/input/eventXX (Kensington ...)` 확인.
3. `learn`으로 4개 코너 버튼 코드 확인 → config의 뒤로/앞으로 버튼 확정.
4. 브라우저에서 뒤로/앞으로 동작 검증.

## 9. GUI 구조 (`ktrackball_gui.py`)

- GTK3(PyGObject) 사용. Wayland 호환 위해 **사용자 권한으로 실행**.
- 권한 필요한 2가지만 `pkexec`로 데몬 스크립트 호출:
  - **버튼 감지**: `learn-once` — 데몬 잠시 중지 → 버튼 1회 입력 캡처 → 이름 출력 → 데몬 복구
  - **저장**: `apply <tmp>` — config 검증 → `/etc/ktrackball/config.toml`(root:root 0644)로 설치 → 데몬 재시작
- 동작 프리셋: 일반클릭/뒤로/앞으로/홈(Alt+Home)/Home키/End키/PageUp·Down/맨위·아래/
  새탭/탭닫기/새로고침/복사/붙여넣기/볼스크롤/정밀모드/사용자키조합/명령실행
- 실행: `python3 /opt/ktrackball/ktrackball_gui.py` 또는 앱 메뉴 "Kensington 트랙볼 설정"

## 10. 보안 (GitHub 공개 관련)

- 비밀정보·자격증명·텔레메트리 없음. PII(사용자명) 제거 완료.
- 데몬은 root 실행(=`/dev/input` 읽기 → 키 입력 감시 가능 / `/dev/uinput` 주입). 모든
  입력 리매퍼(input-remapper, keyd)와 동일한 본질적 권한 → 신뢰하는 코드만 실행.
- `/etc/ktrackball/config.toml`은 **root:root 0644 유지**. 비root 쓰기 가능하면 로컬
  사용자가 `command` 액션을 주입해 권한 상승 가능 → install/`apply`가 소유권 강제.
- GUI는 root로 안 돌고 `pkexec`로 매번 **인증 요구**. 무인증 polkit 규칙 추가 금지.
